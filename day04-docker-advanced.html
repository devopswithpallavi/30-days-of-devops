01
Multi-Stage Builds

A technique where one Dockerfile has multiple FROM instructions — each starts a new build stage. Build your app in one stage (with all build tools) and copy only the final output into a small, clean production image.

💡 Why it matters: A Node.js app with all dev dependencies = 800MB+. With multi-stage build = ~80MB. 10x smaller. Faster to pull, less attack surface, cheaper storage.
❌ Without Multi-Stage
All build tools in final image. node_modules, compilers included. Image: 800MB–2GB. More attack surface. Slow pipelines.
✅ With Multi-Stage
Only final binary in image. No build tools in production. Image: 50–150MB. Minimal attack surface. Fast deploys.

Node.js Multi-Stage Example

 Dockerfile
# Stage 1: Builder
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .                 RUN npm run build
# Stage 2: Production
FROM node:20-alpine AS production
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER node               EXPOSE 3000
CMD ["node", "dist/index.js"]

Go Multi-Stage (ultra-small ~5MB)

 Dockerfile
FROM golang:1.21 AS builder
WORKDIR /app           COPY . .
RUN CGO_ENABLED=0 go build -o myapp .
FROM scratch              # zero OS!
COPY --from=builder /app/myapp /myapp
CMD ["/myapp"]           # ~5MB final image 🔥
02
Image Optimization

The base image is the biggest factor in image size. Always prefer slim or alpine variants. Never use full OS images in production.

Base ImageSizeUse When
node:20~1GBNever use in production!
node:20-slim~240MBNeed some system libraries
node:20-alpine~50MBMost Node.js apps ✅
python:3.11-slim~130MBPython apps ✅
scratch~0MBGo/Rust static binaries ✅
distroless~20MBMaximum security production ✅

Minimize Layers — combine RUN commands

 ❌ Bad — 3 layers
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*
 ✅ Good — 1 layer
RUN apt-get update && \
    apt-get install -y curl && \
    rm -rf /var/lib/apt/lists/*

.dockerignore — exclude from build context

 .dockerignore
node_modules .git .env *.log
README.md __pycache__ tests/
03
Layer Caching

Docker caches every layer. If a layer hasn't changed, Docker reuses the cache — builds become extremely fast. Order of Dockerfile instructions matters hugely.

🔥 Golden Rule: Put things that change rarely at the TOP. Things that change often at the BOTTOM. Maximize cache hits.
❌ Cache-busting
1. COPY . . ← changes every time
2. RUN npm install ← always re-runs

Every build reinstalls all deps.
✅ Cache-friendly
1. COPY package.json ← rarely changes
2. RUN npm install ← cached! ✅
3. COPY . . ← only this rebuilds
04
Docker Security

Security = reducing attack surface + principle of least privilege. Most vulnerabilities come from running as root, storing secrets in images, or using bloated base images.

Never run as root

 Non-root user in Dockerfile
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser              CMD ["node", "app.js"]

Never store secrets in images

 ❌ Never do this
ENV DB_PASSWORD=mysecret
# baked into image forever!
 ✅ Inject at runtime
$ docker run -e DB_PASSWORD=$SECRET myapp

Docker Scout — Vulnerability Scanning

 Docker Scout
$ docker scout cves myapp:latest          # scan CVEs
$ docker scout quickview myapp:latest     # quick summary
$ docker scout compare myapp:v1 myapp:v2  # compare versions
Non-root user
Use USER instruction. Limits blast radius if container is compromised.
Scan images
Use docker scout cves to find known vulnerabilities before deploying.
Pin versions
Use node:20.11-alpine not node:latest — reproducible, predictable builds.
--read-only
Run containers with read-only filesystem to prevent unwanted writes.
Distroless
Use gcr.io/distroless images — no shell, no package manager, minimal surface.
05
Private Registry

A registry stores and distributes Docker images. Docker Hub is public. In production, companies use private registries to keep images secure and internal.

Docker Hub
Default public registry. Free tier with pull limits. Good for open-source.
AWS ECR
Amazon Elastic Container Registry. Integrates with ECS & EKS. Most popular on AWS.
GitHub GHCR
GitHub Container Registry. Free for public repos. Works with GitHub Actions CI/CD.
Azure ACR
Azure Container Registry. Best for Azure-based workloads.
Google GCR
Google Artifact Registry. Integrates with GKE and Cloud Build.
Self-hosted
Run your own with registry:2 Docker image. Full control, no external dep.

Push to registries

 Push commands
# Docker Hub
$ docker login                                                                          
$ docker tag myapp:v1 username/myapp:v1        $ docker push username/myapp:v1
# GitHub GHCR
$ echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin
$ docker tag myapp:v1 ghcr.io/username/myapp:v1  $ docker push ghcr.io/username/myapp:v1
06
Healthchecks

Docker periodically runs a command inside the container to check if it's healthy. If the check fails, Docker marks the container as unhealthy. Kubernetes uses this to auto-restart broken containers.

💡 Without healthchecks, a container that started but isn't serving traffic still appears "running" — you won't know it's broken until users complain.
 HEALTHCHECK in Dockerfile
# Check every 30s, timeout 10s, 3 retries before unhealthy
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
# For PostgreSQL
HEALTHCHECK --interval=10s --timeout=5s \
  CMD pg_isready -U postgres || exit 1
07
Resource Limits

Without limits, one misbehaving container can consume all CPU/memory on the host and crash everything else. Always set limits in production for fair resource sharing.

 Resource limits — CLI & Compose
# CLI flags
$ docker run --memory="512m" --cpus="1.5" myapp
# Docker Compose
services:
  web:
    deploy:
      resources:
        limits:            memory: 512M   cpus: '1.5'
        reservations:   memory: 256M   cpus: '0.5'

Production Best Practices
01 Multi-stage
Smaller, faster, more secure images — remove all build tools from production.
02 Alpine/slim base
Use node:20-alpine not node:20. Smaller size, less attack surface.
03 Non-root user
Add USER appuser in Dockerfile. Principle of least privilege.
04 .dockerignore
Don't include secrets, node_modules, .git, logs in build context.
05 Pin versions
Use node:20.11-alpine not node:latest — reproducible builds.
06 Layer order
Copy deps files first → install → copy code. Maximize cache hits.
07 One process
One process per container. Easy to scale, monitor, and restart independently.
08 HEALTHCHECK
Auto-detect unhealthy containers. Required for Kubernetes liveness probes.
09 Resource limits
Set memory and CPU limits. Prevent one container starving others.
10 Scan images
Run docker scout cves before every production deploy.
11 No secrets
Never bake secrets in ENV instructions. Inject at runtime or use secrets manager.
12 Named volumes
Persist data across container restarts. Never rely on container layer for data.
🚀 Checklist: Multi-stage ✓ Non-root ✓ .dockerignore ✓ Pinned versions ✓ Resource limits ✓ Healthcheck ✓ Security scan ✓ Private registry ✓
